# PLAN-0006: harness forkability cleanup

## Goal
Make the harness actually work for template forkers, and remove the PLAN-0004 additions that are either broken for forkers or add more noise than signal. Specifically: de-hardcode the `nl/jinsoo/template/` package path, narrow `check_plan_progress` to stop sibling-plan false positives, honestly relabel PostToolUse-hook markers as `Nudge:` (PostToolUse is structurally non-blocking per the Anthropic hooks reference), delete the four weakest checks shipped by PLAN-0004, and relax the rule text where enforcement is already owned by a stronger JVM-level gate.

## Non-goals
- Not deleting the PLAN-0003 style-scan bundle, Lombok deny-list, test naming, deprecated test-API, or test-reminder hooks — those are Bucket A (high signal, cheap) and earn their keep.
- Not touching the exec-plan *format* itself (session blocks, decision log, execution notes, ≤12 steps, required sections). The format is convergent with industry practice (OpenAI Codex cookbook, Kaushik Gopal, Augment Code). Only the PostToolUse *enforcement* of that format is being pared back.
- Not rewriting hooks in Python / Node / another language. Shell is proportional for regex scanning.
- Not replacing the `audit` agent or its invocation pattern.
- Not touching PLAN-0005 (safety-guard hooks) — that plan is still active and out of scope here.
- Not changing `full-check`, ArchUnit rules, Spotless, NullAway, or `ApplicationModules.verify()`. The JVM-level enforcement tier is the source of truth and is untouched.

## Approach
The six fixes from the critique group naturally into three sessions:

1. **Block 1 — Ship-stopping Priority 1 fixes.** De-hardcoding (Fix 1), plan-progress narrowing (Fix 2), and honest `Nudge:` relabeling (Fix 3) are all small, independent, and needed regardless of what happens to the bigger structural fixes. They ship together because they share a single `test-hooks` run.

2. **Block 2 — Delete Bucket B/C plan-file diff hooks.** `check_decision_log_append` and `check_execution_notes_after_block` are unactionable nudges that depend on judgment the regex cannot make. Delete both. In exchange, extend `check_plan_progress` with a single pre-finalization check that fires only when all steps are `[x]` and verifies decision log + execution notes are non-empty. Runner's `before/` git-diff extension can then be removed because nothing else uses it. Regression-first: add a failing fixture for the pre-finalization check *before* extending `check_plan_progress`.

3. **Block 3 — Delete module contract sync, reshape the contract format, relax rule text, close.** `check_module_contract_sync` automates exactly the half of the module contract that `ApplicationModules.verify()` + `@ApplicationModule(allowedDependencies)` already enforce as the Spring Modulith source of truth (per `docs.spring.io/spring-modulith/reference/fundamentals.html`). Delete the check, delete the library, drop the Public API + Allowed dependencies markdown tables, keep the behavioral half (Consumer surface, Behavioral notes, Owned resources) as informational-only. Relax `rest.md` § "OpenAPI annotations (mandatory)" to match what the hook actually enforces + point at `check-openapi-drift` for `operationId` uniqueness. Finalize with full-check, audit, and move to completed.

The whole plan is NET-negative on code (~660 shell lines deleted, ~15 fixtures deleted, ~100 lines of rule/contract text simplified) minus ~60 lines added for the narrowed `check_plan_progress`, the pre-finalization check, and the `harness.env` config.

## Risks & mitigations

| Risk | Mitigation |
|---|---|
| `harness.env` rename breaks sourcing in environments where the file is not present | Use `"${HARNESS_ROOT_PACKAGE:-nl/jinsoo/template}"` fallback in every call site. Missing `harness.env` degrades to the original hardcoded behavior, not a hard failure. Document the convention in both the root `README.md` and in a comment at the top of `harness.env` itself. |
| Deleting `check_module_contract_sync` removes the one mechanism that catches new public types added without a contract row — module contracts could now drift silently | This is exactly the point: the Public API + Allowed dependencies half of the contract duplicated `ApplicationModules.verify()` and was the wrong half to hook in the first place. The *unique* value of module contracts (Consumer surface, Behavioral notes) was never checked by the hook. After this plan the audit agent remains as a periodic-sweep safety net for behavioral drift, and `ApplicationModules.verify()` catches allowed-dependency drift at test time. |
| Narrowing `check_plan_progress` misses a legitimate "no plan in progress" warning when the user edits a Java file after a context reset and forgets to pick up the plan | The narrowed logic still emits the warning — it just emits *once* instead of *once per active plan*. The single warning is louder per-occurrence (not drowned in duplicates), which should be equally effective or better. |
| Relaxing the `rest.md` OpenAPI clause makes template users think `operationId` is optional | The clause explicitly states `check-openapi-drift` is the authoritative gate for `operationId` uniqueness. Forgetting it becomes a full-check failure, not a silent omission. Rule text now matches enforcement reality. |
| Deleting PLAN-0004 Block 2 output partially undoes an audit-passed plan and could set a precedent of undoing completed work | Decision-logged: PLAN-0006 is explicitly a follow-up cleanup, not a retroactive reversal. PLAN-0004 shipped what it promised; this plan reshapes the approach based on post-hoc review against current Spring Modulith guidance and the Claude Code hooks reference. The reshape is documented as a directional pivot, not a bug fix. |
| The runner `before/` and `files/` extensions are removed, and a future check needs them | Both extensions are small (~15 lines each). Re-adding them when a future check genuinely needs git-diff or multi-file seeding is cheap. Removing them now keeps the runner surface minimal for the current check set. |

## Tech debt introduced
- **Module contract drift is no longer hook-enforced.** After this plan, if a contributor adds a new public type to `nl.jinsoo.template.notepad` and forgets to update `notepad.md`, only the audit agent (run periodically) will catch it. This is acceptable because the audit agent exists specifically for behavioral-contract drift and `ApplicationModules.verify()` catches allowed-dependency drift at test time — but it does move drift detection from edit-time to periodic sweeps.
- **`before/` fixture-runner extension is deleted; `files/` extension survives Block 3.** The plan originally intended to delete both; `before/` is gone (no remaining callers) but `files/` stays because six `check_openapi_annotations_present` fixtures still seed controller source files via the extension. A future plan can either (a) migrate the openapi fixtures to `content` via a new multi-file encoding, (b) add a lighter-weight runner primitive, or (c) leave `files/` in place indefinitely. If a future PostToolUse hook genuinely needs git-diff fixture seeding, the `before/` extension will need to be re-added (~15 lines). Documented here so a future plan doesn't re-derive it from scratch.
- **`scripts/harness/lib/harness.env` is no longer sourced by any library.** The file still exists on disk as the durable fork-time rename point and forker convention, and Block 1's README entry still documents it as "first thing to edit after forking." But after Step 7 deleted `check_module_contract_sync` — the only consumer of `${HARNESS_ROOT_PACKAGE}` — the `harness.env` sourcing block in `hook-checks.sh` became dead code and was removed. A future PostToolUse check that needs the root package will need to re-add the sourcing block at the top of `hook-checks.sh`. Not urgent — no known near-term check requires it — but recorded here so a future plan author knows the infrastructure exists half-wired.
