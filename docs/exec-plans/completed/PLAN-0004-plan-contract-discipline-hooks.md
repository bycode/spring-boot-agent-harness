# PLAN-0004: plan contract discipline hooks

## Goal
Ship the cross-file, content-aware hooks that enforce plan-file discipline (decision log append, execution notes after each block, session block structure for plans >5 steps), module contract drift detection (public API table vs. actual root-package types; `@ApplicationModule` allowed dependencies vs. markdown), and OpenAPI annotation presence on new REST controllers.

## Non-goals
- Per-file regex hooks (test naming, deprecated APIs, Lombok, Tier-B bundle) — PLAN-0003 owns those and must ship first
- Git pre-commit guards and subprocess output scrubbing — PLAN-0005
- OpenAPI spec drift between app and `docs/generated/openapi.json` — already enforced by `scripts/harness/check-openapi-drift`; the hook here catches *missing* annotations on newly added endpoints before drift-check would notice
- Parsing Java source with a real parser — use shell grep + simple line scanning. A full AST parser is out-of-scope over-engineering for the drift patterns we care about

## Approach
1. **Plan-file diff hooks first (Block 1)**: `check_decision_log_append`, `check_execution_notes_after_block`, and `check_session_block_structure` all operate on `docs/exec-plans/active/*.md`. They use `git diff HEAD -- <file>` (or in-memory cached pre-edit state) to detect the *transition* a file just made (e.g., a step flipping from `[ ]` to `[x]`) and check whether a related section grew. Ship these as a group because they share the plan-file parsing helpers.
2. **Module contract sync (Block 2)**: the biggest single-rule addition. Split into three steps: (a) build a shared parsing library that enumerates public root-package types and parses the markdown contract's "Public API" + "Allowed dependencies" sections, (b) wire it into a hook function, (c) extend to cross-check `@ApplicationModule(allowedDependencies = ...)` in `package-info.java`.
3. **REST annotation presence (Block 3)**: `check_openapi_annotations_present` is smaller but complements Block 2 well (both guard public contracts). Finalize with wiring, docs, full-check, audit.

## Risks & mitigations

| Risk | Mitigation |
|---|---|
| Shell-based markdown table parsing is fragile and breaks on unusual formatting | Step 4 defines a strict contract format (header row, separator row, data rows with fixed column count) matching MODULE-TEMPLATE.md; contracts that don't match get a "can't parse" warning, not a silent pass. If bash proves too fragile, fall back to Python for the table parse (decision logged). |
| `git diff HEAD` behavior varies depending on whether the file was staged or not; hook sees stale state | Use `git diff HEAD -- "$FILE"` specifically (works for both staged and unstaged), with fallback to `git show HEAD:"$FILE"` for the pre-state. Fixture runner simulates both cases. |
| Public-type enumeration grep misses subtle definitions (e.g., multi-line sealed declarations) | Fixture coverage includes multi-line declarations; if grep proves insufficient, escalate to a small awk state machine (decision logged). |
| Module contract sync warns on legitimate refactors mid-edit (e.g., just renamed a type, haven't updated markdown yet) | Non-blocking warning, not an error. The whole point is to remind the agent that the contract needs updating — that's a helpful interruption, not a blocker. |
| `check_openapi_annotations_present()` preceding-line heuristic (20-line lookback) misses annotations on large methods | 20-line window matches the style most controllers actually use; if it proves too narrow, widen or switch to a simple "between the previous `}` and the current `@Mapping`" scan. |
| Hook runs on every plan-file edit even for trivial whitespace changes | `check_decision_log_append` and `check_execution_notes_after_block` compute a before/after diff — if nothing meaningful changed in the Steps section, they emit nothing. |

## Decision log

| Date | Decision | Rationale |
|---|---|---|
| 2026-04-10 | Runner `before` file support: optional per-fixture file; runner only `git init`s `$tmp_root` when a fixture actually ships a `before` file | Keeps the 49 pre-existing fixtures untouched and avoids forcing every fixture into a git-aware path. The extension is strictly additive |
| 2026-04-10 | Use `substr($0, 1, 4)` prefix compares in `_last_checkbox_per_block` instead of `match($0, regex, m)` | Default `awk` in this sandbox is `mawk`, which does not implement gawk's 3-arg `match` capture API. Substr stays portable and the checkbox prefixes are fixed-width |
| 2026-04-10 | Match blocks between pre/post-state by full `### Block N — label` header line | Simple and robust when headers do not change. Documented limitation: a rename of the header in the same diff that closes its last checkbox will under-warn. Non-blocking |
| 2026-04-10 | Scoped `-Warning:` negatives on plan-file fixtures where `check_plan_progress` also fires | Plan fixtures under `docs/exec-plans/active/` necessarily trip the progress check; scoped negatives isolate the check under test without brittle `EMPTY` assertions |
| 2026-04-10 | Implement `module_public_types` with a token-based awk scanner rather than a flat `grep -E` | A flat grep misfires on `public String name;`, `public static final long ID`, and `public void record(...)` because `record` is also a kind keyword. The scanner tracks `saw_public` + modifier run and only emits when it reaches a real kind keyword (`record`/`class`/`interface`/`enum`). It also handles multi-line `public sealed\ninterface Foo` via a `pending_kind` continuation |
| 2026-04-10 | Annotation-name matching in `package_info_allowed_dependencies` walks backwards from `ApplicationModule` to find the preceding `@` | Annotation may be fully qualified (`@org.springframework.modulith.ApplicationModule(...)`). Matching a literal `@ApplicationModule` missed the FQN form. Walking back over `[A-Za-z0-9_.]` and asserting the preceding char is `@` handles both forms without regex capture groups |
| 2026-04-10 | Standalone lib tests live in `scripts/harness/tests/module-contract-lib-test` and are invoked from the top of `scripts/harness/test-hooks` before any fixture runs | Keeps the fixture runner focused on hook-dispatcher output assertions (what `post-tool-hook` emits) while still exercising `module-contract.sh` helpers directly with synthetic inputs. A broken helper fails fast before 60+ fixture tests start, shortening the feedback loop. The alternative (a new fixture-runner "lib-test mode") would have forced significant changes to the case.json/expect.txt contract for no clear win |
| 2026-04-10 | Extend fixture runner with optional per-fixture `files/` directory whose contents are copied verbatim into `$tmp_root` | Module contract fixtures need to seed multiple files at once (module source + contract markdown). Keeping the existing single-file `content` mechanism backwards-compatible means the pre-existing 61 fixtures are untouched. `files/` is strictly additive |
| 2026-04-10 | `check_openapi_annotations_present` does NOT enforce `operationId = "..."` even though `rest.md` § "OpenAPI annotations (mandatory)" says it's mandatory | `rest.md` § "Accepted when" explicitly exempts the `notepad` reference module from `operationId`, and regex-matching `operationId` across multi-line `@Operation(...)` annotations is fragile. The existing notepad controller uses `@Operation(summary = "...")` on every method — the check must stay silent on it. The check still guards the high-signal cases: `@RestController` without `@Tag`, and `@<Verb>Mapping` with no `@Operation` anywhere in the method's annotation block |
| 2026-04-10 | `check_openapi_annotations_present` does NOT re-implement the URI-prefix (`/api/`) rule | `check_style_scan` already flags `@<Verb>Mapping("path")` where path does not start with `/api/`. Duplicating the rule would double-warn. The step's original brief mentioned URI prefix as part of scope; that's out of scope for this check by design, documented here |
| 2026-04-10 | Abandoned the "preceding 20 lines" lookback approach in favor of an annotation-block accumulator | Original awk scanner used a ring buffer to look 20 lines back from each `@<Verb>Mapping` line for an `@Operation(`. That broke when `@Operation` appears AFTER `@<Verb>Mapping` (valid Java annotation ordering). The accumulator collects all annotations between the last non-annotation line and the next method signature, then emits a warning only if the block has `@<Verb>Mapping` but no `@Operation`. More robust, and matches the semantic "method's annotation block" intent of the rule |
| 2026-04-10 | Mapping-annotation regex uses `@(Get\|Post\|Put\|Delete\|Patch)Mapping([[:space:]]\|\(\|$)` instead of the simpler `\b` word boundary | `mawk` does not support `\b` in regex (same class of limitation as the `match(..., array)` capture issue called out in Block 1). Explicit post-anchor alternation over whitespace, `(`, and end-of-line covers every real-world mapping annotation form without depending on a gawk extension |
| 2026-04-10 | `Enforcement:` lines placed at end-of-clause (before the next `##` heading) rather than immediately inside each numbered rule | Keeps the numbered rules readable as policy and groups the hook reference as a single metadata trailer per section. This matches the PLAN-0003 style for `check_style_scan` / `check_lombok_denylist` Enforcement: lines already present in `code-style.md`, `virtual-threads.md`, and `rest.md` |
| 2026-04-10 | Manual exercise used `public record PublicProbe(long value) {}` rather than a class | Records are the cheapest way to declare a public type in Java 25 and trigger `module_public_types` without pulling in framework imports. The scanner treats `record` as a kind keyword (see Block 2 decision log), so the throwaway type is indistinguishable from a real domain record for drift purposes |
