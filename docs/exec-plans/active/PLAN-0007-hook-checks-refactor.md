# PLAN-0007: hook checks refactor

## Goal
Fix every finding from the PLAN-0006 close-out `/simplify` review: eliminate the dead `harness.env` infrastructure, fuse redundant awk/grep passes in `check_plan_progress` and `check_style_scan`, extract shared helpers for repeated path-guard / import / annotation-list patterns, shrink the per-tool-call jq overhead in the dispatcher, and strip orphan locals + outdated docstring references. Pure refactor — same inputs must produce byte-identical warning text.

## Non-goals
- Not changing the dispatcher contract (exported `TOOL/FILE/CMD/INPUT/REPO_ROOT` globals). `/simplify` LOW finding #10 is deferred — it would touch every check function's signature and the fixture runner's case.json format, out of scope for a quality cleanup plan.
- Not unifying `scripts/harness/new-module` heredoc structure. `/simplify` LOW finding #14 — marginal value, defer until the next edit touches the file.
- Not extending `scripts/harness/test-hooks` expect.txt DSL beyond its three keywords. `/simplify` LOW finding #16 — defer until a fourth keyword is needed.
- Not adding new hook checks. This plan refactors existing checks only.
- Not touching PLAN-0005 (safety-guard hooks, still active in `docs/exec-plans/active/`).
- Not modifying PLAN-0006 itself. Its completed file stays as historical record; reversal decisions (e.g., deleting `harness.env`) are logged in THIS plan's decision log.
- Not changing `.claude/rules/*.md` prose except where a surviving reference to `harness.env` or a deleted-in-this-plan construct needs a consistency update.
- Not touching ArchUnit, Spotless, NullAway, `ApplicationModules.verify()`, JaCoCo, `check-openapi-drift` (JVM-level gates).
- Not adding NEW failing regression fixtures. Existing fixtures serve as the regression contract — see decision log.

## Approach
The 16 `/simplify` findings fall into four families: (1) dead code removal, (2) hot-path efficiency in `check_plan_progress` and `check_style_scan`, (3) helper extraction across repeated patterns, (4) small dispatcher cleanups. Group by risk and file-locality into 6 steps across 2 blocks.

**Block 1 targets the hot path.** Delete dead infrastructure first (Step 1, zero risk). Then fuse the two biggest offenders: `check_plan_progress`'s 1-awk + 3-grep + 2-awk-reread pattern collapses to one awk per plan (Step 2), and `check_style_scan`'s 17 sequential greps collapse to one awk pass (Step 3). These are the highest-impact efficiency wins — both run on every Edit/Write from the agent.

**Block 2 is DRY-up + polish + close.** Introduce three shared helpers (`_guard_edit_write_java`, `_has_import`, `_grep_annotation_at_line_start`) and apply them across all surviving checks (Step 4). Clean up the three jq calls in the dispatcher, remove a dead local + outdated PLAN reference (Step 5). Verify with full-check + audit + close (Step 6).

**Regression strategy:** every step is a pure refactor — no behavior change. The existing 62 fixtures under `scripts/harness/tests/hooks/*` are the regression contract: same input → same warning text. For each step that touches a check function, the acceptance gate is "test-hooks stays green with the existing fixture set." If Step 3's audit of `check_style_scan` finds a pattern with no fixture coverage, we add the fixture BEFORE the refactor (regression-first at the "find and fill coverage gap" level, not at the "introduce a bug to test" level).

**Expected line count delta:** baseline is 927 shell lines post-PLAN-0006 (hook-checks.sh 677 + post-tool-hook 45 + test-hooks 205). Target after this plan: ≤ 850 (a further ~80 lines removed). Step 1 removes ~30 lines (harness.env file + test-hooks injection block). Steps 2 and 3 shrink hot-path code by ~40 lines through fusion. Step 4 DRYs out ~30 lines via helpers but adds ~15 lines of helper definitions, net ~15 lines. Step 5 is ~10 lines net.

## Risks & mitigations

| Risk | Mitigation |
|---|---|
| Fusing `check_plan_progress` awk passes introduces a subtle counting bug (e.g., off-by-one on section boundaries, missed `[~]` with leading whitespace) | Coverage audit in Step 2 verifies every branch has a fixture; existing fixtures assert byte-identical warning text via `+substring` matchers; run `test-hooks` after the awk fusion and before the narrowing cache to isolate which change (if any) regresses a fixture. |
| Rewriting `check_style_scan` as one awk misses a pattern that only had partial fixture coverage | Step 3 explicitly inventories every pattern and audits fixture coverage BEFORE refactoring. Any uncovered pattern gets a fixture FIRST. If the inventory reveals a pattern with NO current fixture, treat it as a latent tech-debt item and add positive + content-negative fixtures in the same step. |
| Helper extraction changes the output format of an existing warning (e.g., `_grep_annotation_at_line_start` joins matches differently than the old loop) | Helpers are designed to produce byte-identical output to the inline code they replace — a helper that changes format is a bug, not a design choice. Run `test-hooks` with full `+substring` assertions to catch any drift. If a fixture needs updating because the old output format was inconsistent, document it in the decision log. |
| Native bash regex (`=~`) has different escape rules than `grep -E` | Double-check every converted regex; bash `=~` uses POSIX ERE by default but some `grep -E` patterns rely on GNU extensions. Prefer alternatives like `case "$CMD" in ... esac` if `=~` proves fragile for a specific pattern. |
| Caching narrowing results via parallel bash indexed arrays has portability issues on older bash (pre-4.0, e.g. macOS default /bin/bash 3.2) | The harness already uses bash 4+ features elsewhere (`local -A`, process substitution). Indexed arrays work in bash 3.2. Use indexed arrays, not associative arrays, to maximize portability. If a portability issue surfaces, fall back to tab-delimited tmp file per plan with `mktemp -p $REPO_ROOT`. |
| The `_simplify` review itself missed something; fixing the 16 findings leaves deeper issues | Accept. This plan is scoped to the findings list, not to "refactor hooks comprehensively." If Step 6's audit surfaces NEW issues, either fix in-place (LOW) or log as follow-up (MEDIUM/HIGH) and leave for a future plan. |
| Step 3's `check_style_scan` awk rewrite is large enough to warrant its own plan | Re-evaluate during execution. If Step 3 grows past ~80 lines of awk or reveals fixture-coverage gaps requiring >3 new fixtures, STOP, mark the step skipped with a decision-log entry, and split into a follow-up plan (`PLAN-0008-style-scan-awk-fusion`). The 12-step hard limit keeps the plan focused. |
